## 1. 개요

### 1.1 배경

**ai-dt.net**(「AI·디지털 교육자료 수업 지원센터」)은 현재 구글 사이트로 운영 중인 교사 대상 AIDT(AI 디지털교과서) 지원 허브다. 협업 편집은 편했으나:

- **반복 구조를 수동 복제**해야 한다 — `/index/feedback/` 아래 출판사별 페이지 12개(금성·길벗·동아·디딤돌·미래엔·비상·씨마스·아이스크림미디어·엔이능률·와이비엠·지학사·천재)가 같은 틀로 병렬 존재. 매뉴얼·소프트웨어·FAQ도 "같은 형식 × N개" 패턴.
- 페이지당 HTML 150~180KB로 무겁고, 봇 접근에 403을 반환해 **SEO 불리**.
- 디자인·레이아웃 자유도 낮음(캐러셀 등 표현 한계). 향후 콘텐츠 대량 추가 시 운영 부담 가중.

노션+커스텀 도메인도 검토했으나 커스텀 도메인이 유료이고(무료 티어 원칙 위배), 레이아웃 한계가 구글 사이트와 유사하며, 운영 모델이 이미 "운영자가 취합"이라 노션의 직접 발행 우위가 소멸 → **자체 구축으로 결정**.

### 1.2 현재 사이트 구조 (2026-07-09 크롤링 실측)

내부 페이지 약 25개, 4개 대분류:

```
/               홈
/index          안내 허브
  /index/AIDT         AIDT 안내
  /index/eduonepass   교육디지털원패스 안내
  /index/manual       매뉴얼
  /index/software     소프트웨어
  /index/faq          FAQ
  /index/feedback     출판사 피드백 허브
    /index/feedback/{출판사} × 12
/class          수업이야기
  /class/curriculum
  /class/중등-ai디지털-교육자료-지원
/study          스스로 학습 안내
/contact        문의 및 장애 지원
```

주요 외부 연결: aidtbook.kr, edupass.neisplus.kr(교육디지털원패스), aidtshow.kr, dge.go.kr, AIDT 튜터링(구글 사이트).

### 1.3 목표

- 반복 콘텐츠를 **파일(데이터) + 템플릿**으로 전환해, 항목 추가 = 파일 1개 추가로 만든다.
- 기존 25개 페이지를 정보 손실 없이 이관하고, 속도·SEO·접근성을 구글 사이트 대비 개선한다.
- 동료 운영자가 git 없이 참여하는 저작 워크플로우(양식 파일 → 운영자 커밋)를 확립한다.
- 무료 티어(Cloudflare Pages)로 전 기능 운영한다.

### 1.4 범위 (In / Out of scope)

**In scope**
- Astro 사이트 골격, 공통 레이아웃(헤더/네비/푸터), 재사용 컴포넌트(카드 그리드, 캐러셀, 영상 임베드, 링크 버튼, 탭, FAQ 아코디언).
- 콘텐츠 컬렉션 4~5종(§2) 스키마 정의 + 기존 콘텐츠 전량 이관.
- 동료용 저작 양식(마크다운 템플릿 + 메타 항목표) 문서.
- 사이트 내 검색(정적 인덱스 기반, 예: Pagefind), OG 메타, sitemap, favicon.
- ai-dt.net 커스텀 도메인 연결.

**Out of scope (현 단계)**
- CMS·어드민 UI·로그인·계정(C3 — 파일 전달 모델로 충분).
- 런타임 DB·서버 기능(문의는 기존 외부 폼/채널 링크 유지).
- 조회수·좋아요 등 집계 기능(필요해지면 별도 plan으로).
- 기존 구글 사이트 URL 경로와의 1:1 리다이렉트 보존(경로 체계를 새로 설계하되, 주요 진입 URL만 `_redirects`로 대응).

### 1.5 비목표 (Non-goals)
- 구글 사이트 디자인의 픽셀 재현 — 정보 구조만 계승하고 디자인은 새로 잡는다.
- 동료가 직접 발행하는 체계 — 운영자 취합 모델이 전제. 추후 수요가 생기면 시트 연동(§6)으로 확장.

## 2. 콘텐츠 모델 (Content Collections)

| 컬렉션 | 대상 | frontmatter 핵심 필드 (초안) |
|---|---|---|
| `pages` | 홈·안내·수업이야기 등 일반 위지윅형 페이지 | `title, section, order, updated` + 마크다운 본문(영상·이미지·캐러셀은 컴포넌트 삽입) |
| `publishers` | 출판사 12개 | `name, subjects[], links{피드백폼, 지원채널, 매뉴얼}, logo, note` — 본문 없이 데이터만으로 페이지 생성 |
| `resources` | 매뉴얼·소프트웨어·도움자료 카드 | `title, category, target(교사/학생), url, thumb, desc, updated` |
| `faq` | FAQ 항목 | `q, category, order` + 본문 = 답변 |
| `cases` | 수업사례 (향후 대량 추가 예상 영역) | `title, school_level, subject, author, date, tags[]` + 본문 |

- 스키마는 `src/content.config.ts`에서 zod로 검증 → 동료가 준 파일의 메타 누락·오타를 빌드가 잡아줌.
- 파일명 규칙: 한글 허용하되 URL slug는 frontmatter `slug` 필드로 명시(한글 URL 회피 여부는 구현 시 결정).

## 3. 사이트 구조 (신규 IA 초안)

> **제약: 1depth 경로 유지.** 기존 1depth 페이지(`/index`, `/class`, `/study`, `/contact`)는 외부에 공유된 진입점이므로 **접속 주소를 그대로 유지**한다. IA 개편은 각 1depth 하위에서만 한다.

```
/                홈 (주요 안내 카드 + 바로가기)
/index/          안내 허브 (유지)
  /index/faq/
  /index/manual/ , /index/software/   → 자료 카드 (resources 컬렉션)
  /index/feedback/                    출판사 허브 (12개 카드 그리드)
  /index/feedback/{slug}/             출판사별 페이지 (publishers 컬렉션)
/class/          수업이야기·수업사례 (유지)
/study/          스스로 학습 안내 (유지)
/contact/        문의 및 장애 지원 (유지)
```

- 2depth 이하도 가급적 기존 경로를 따르되, 변경이 불가피한 곳(한글 경로 등)만 `_redirects`로 대응.
- 신규 섹션이 필요해지면 1depth를 추가하는 방식으로 확장(기존 경로는 불변).

## 4. 기술 구성

- **Astro** (content collections, 정적 빌드) — 클라이언트 JS 기본 0, 캐러셀 등 필요한 곳만 `<script>`.
- **스타일**: Tailwind CSS (워크스페이스 통상 스택과 일치).
- **검색**: Pagefind (빌드 후 정적 인덱스, 무료·서버리스).
- **배포**: GitHub 레포 → Cloudflare Pages 자동 빌드(`astro build`, 산출물 `dist/`).

### 4.1 디자인

- 구글 사이트 디자인을 복제하지 않고 **새로 설계**한다. 방향: 교육청 지원센터 톤의 밝고 단정한 카드 중심 레이아웃, 모바일 대응(교사 상당수가 폰으로 접속), KWCAG 접근성 기본 준수.
- 절차: P1에서 **홈 + 출판사 페이지 2종의 시안**(실제 Astro 페이지)을 먼저 만들어 `*.pages.dev`로 검수받고, 확정된 토큰(색·타이포·컴포넌트)을 전 페이지에 전개한다. 별도 디자인 툴 없이 코드 시안으로 진행.
- 재사용 컴포넌트가 곧 디자인 시스템: 카드 그리드 / 캐러셀 / 영상 임베드 / 링크 버튼 / 탭 / 아코디언 / 안내 배너.

### 4.2 영상·이미지 업로드 방식

| 자산 | 방식 | 비고 |
|---|---|---|
| **영상** | **유튜브 업로드 → 임베드 컴포넌트** (자체 호스팅 안 함) | 마크다운에 `<YouTube id="..." />` 한 줄. 성능 위해 lite-embed(클릭 시 로드) 방식. 기존 구글 사이트 영상도 대부분 유튜브 원본이므로 ID만 옮기면 됨 |
| **이미지** | **레포 내 `src/assets/` 커밋** + Astro 빌드 타임 최적화(WebP 변환·리사이즈·lazy) | 콘텐츠 파일 옆에 이미지를 같이 두는 co-location 구조. 동료는 원본 이미지를 콘텐츠와 함께 전달하면 운영자가 폴더에 넣음 |
| **대용량 파일**(HWP·PDF 배포자료 등) | 1차: 기존 구글 드라이브 공유 링크 유지. 용량·안정성 필요 시 **R2 public 버킷**(무료 10GB, egress 무료) | 현 단계 R2 미도입, 필요 시점에 추가 |

- 업로드 절차 자체가 별도로 없음 — **콘텐츠 파일과 같은 커밋에 자산을 포함**하면 배포 시 함께 나간다. Pages는 파일 25MB/개 제한이 있으므로 그 이상은 유튜브(영상)·드라이브/R2(파일)로 우회.
- 이관 시 구글 CDN에 있는 기존 이미지는 전량 다운로드해 레포에 보관(§7 리스크 대응).

## 5. 단계별 계획

| 단계 | 내용 | 산출물 |
|---|---|---|
| **P1 골격+시안** | Astro 프로젝트 생성, Pages 배포 연결, 홈·출판사 페이지 2종 디자인 시안(§4.1) | `*.pages.dev` 시안 검수 |
| **P2 반복 콘텐츠** | `publishers` 스키마 + 12개 데이터화 + 템플릿, `resources`·`faq` 이관 | 효과 최대 영역 우선 검증 |
| **P3 일반 페이지** | 홈·안내·수업이야기·스스로학습·문의 이관, 임베드 전수 점검(C6) | 콘텐츠 완전 이관 |
| **P4 마감** | 검색(Pagefind)·OG·sitemap·접근성 점검, 동료용 저작 양식 문서 작성 | 저작 가이드 docs |
| **P5 전환** | 관계자 검수 → ai-dt.net DNS 전환, `_redirects` 적용, 구글 사이트 보관 처리 | 정식 오픈 |

## 6. 협업 워크플로우 (동료 2~3인)

1. 동료는 **저작 양식**(P4 산출물: 마크다운 템플릿 또는 "제목/분류/링크/본문" 항목표)으로 작성 → 구글 드라이브/메일 전달.
2. 운영자가 `src/content/` 해당 컬렉션에 파일 추가 → 커밋·푸시 → 자동 배포.
3. 빌드 실패(메타 누락 등)는 스키마 검증이 잡음 → 운영자가 보정.
4. **확장 옵션(현 단계 미구현)**: 메타 입력을 구글 시트로 받고 빌드 시 시트를 읽는 반자동화 — edu-kit 패턴 재사용 가능.

## 7. 리스크

| 리스크 | 대응 |
|---|---|
| 구글 종속 임베드(Apps Script·Drive 뷰어)가 이관 후 깨짐 | P3에서 전수 인벤토리 → iframe 유지 가능 여부 확인, 불가 시 자체 재구현 또는 링크 대체 |
| 구글 사이트 내 이미지·파일 원본 소실 (구글 CDN 종속) | 이관 시 원본을 전부 레포/드라이브로 내려받아 보관 |
| 한글 경로 유입 링크(외부 문서·공문에 박힌 구 URL) | 전환 전 주요 유입 경로 파악, `_redirects` 대응. 대응 불가분은 구글 사이트에 안내 배너 |
| 도메인 전환 실수로 접속 불가 | DNS TTL 낮춘 뒤 전환, 즉시 롤백 경로 확보 |

## 8. 워크스페이스 규칙 적용

- 커밋: Conventional Commits, 한국어 제목 허용.
- 사용자 대상 텍스트 한국어. 개인정보 수집 없음(로그인 없음).
- 구현 진행 시 이 문서 갱신 + `STATUS.md` 생성·유지, 오픈 후 CHANGELOG 시작.
- 최종 안내 URL은 ai-dt.net (`*.pages.dev`는 검수용).
